# Visualization Audit

## Current Surfaces
| Surface | Current Role | Theory Concern |
| --- | --- | --- |
| Card back/front | Main visual object. | Uniform class is good, but mechanic state needs overlays/tells. |
| Action dock | Power buttons and utility controls. | Needs clearer grouping for immediate, armed, passive, and resolve actions. |
| HUD | Run status, score, lives, floor facts. | Build and objective state can be too compressed. |
| Floor-clear modal | Rewards, next route, relic/shop decisions. | Needs stronger system continuity from floor to floor. |
| Relic draft | Build choices. | Archetype tags exist but picks may not feel transformative. |
| Inventory | Run snapshot. | Could become build dashboard, but must avoid fake interactivity. |
| Codex | Knowledge base. | Must explain any new talent/trait/card taxonomy. |
| Tooltips/a11y labels | Precise rule explanation. | Must not lag behind new systems. |

## Missing Visual Languages
- Talent/trait identity badges.
- Build archetype progress or "you are becoming X" readout.
- Card-family icons that stay readable on small screens.
- Power consequence badges: spends charge, arms mode, breaks perfect, no score, forfeits pickup.
- Mutator floor intro and on-board ambient tell.
- Objective progress that feels tied to current card actions.
- Post-run explanation of what a build did.

## Visualization Rule
Every mechanic must have all five layers before implementation tasks are created:

1. Board tell.
2. HUD/action state.
3. Tooltip or label.
4. Codex entry.
5. Result feedback when it triggers.
