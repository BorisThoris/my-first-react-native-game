# REF-003: Unit tests for endless floor mutator schedule
> **Rollup:** Addressed in codebase per [COMPLETION.md](./COMPLETION.md) (evidence row **001–010**).


## Problem

`src/shared/floor-mutator-schedule.ts` owns pure logic for endless mode: `FLOOR_SCHEDULE_RULES_VERSION`, `usesEndlessFloorSchedule`, and `pickFloorScheduleEntry` (fixed cycle of mutators + `floorTag`, seeded optional `distraction_channel` on some boss floors). That behavior is easy to break when editing the cycle or RNG wiring, but regressions are unlikely to be caught by a narrow failure in existing suites—there is no dedicated Vitest file, unlike `mutators.test.ts` for catalog metadata. [GAMEPLAY_POLISH_AND_GAPS.md](../gameplay/GAMEPLAY_POLISH_AND_GAPS.md) (§16) records this as an explicit test gap.

## Proposed work

Add **`src/shared/floor-mutator-schedule.test.ts`** (Vitest, same style as `mutators.test.ts`) with focused cases:

1. **Gates:** non-`endless` modes and `rulesVersion < FLOOR_SCHEDULE_RULES_VERSION` yield `{ mutators: [], floorTag: 'normal' }`; `usesEndlessFloorSchedule` matches `gameMode === 'endless' && rulesVersion >= FLOOR_SCHEDULE_RULES_VERSION`.
2. **Cycle:** for `endless` + qualifying rules version, assert mutators and `floorTag` for a few known `level` values (e.g. first floor, a breather index, a boss index from the current `ENDLESS_FLOOR_CYCLE`).
3. **Wrap:** `level` and `level + k * cycleLength` produce the same entry for fixed `runSeed` / `rulesVersion`.
4. **Boss `distraction_channel` branch:** at least one deterministic `(runSeed, rulesVersion, level)` where the base entry is `boss` and the first RNG draw is `< 0.25`, asserting the result appends `distraction_channel` without duplicates and does not exceed three mutators; plus a case where the draw is `>= 0.25` (or boss already includes that mutator) so the base list is returned unchanged.
