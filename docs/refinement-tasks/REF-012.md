# REF-012 — MainMenu + ChooseYourPathScreen: hub shell fit contract (single source of truth)

| Field | Value |
|--------|--------|
| **ID** | REF-012 |
| **Category** | Shell / viewport fit |
| **Priority** | P2 |

| **Status** | Addressed — hub shell fit (`useFitShellZoom`, `hubShellFit`); see [VIEWPORT_FIT_UI.md](../VIEWPORT_FIT_UI.md) + [COMPLETION.md](./COMPLETION.md) (**011–020**). |

## Summary

Both primary **hub** screens use [`useFitShellZoom`](../../src/renderer/hooks/useFitShellZoom.ts) with **parallel viewport predicates** and **different default padding** (12 vs 14 CSS px). Treat this pair as one shell family: **centralize fit-padding selection** (and optionally the shared “short desktop / touch compact” flags) so menu ↔ choose-path transitions stay visually consistent under resize and future breakpoint edits.

## Evidence (current)

- **Routing:** [`App.tsx`](../../src/renderer/App.tsx) renders `MainMenu` inside `showMenuShell` when `view === 'menu'`; `ChooseYourPathScreen` mounts when `hydrated && view === 'modeSelect'`. The menu **layer unmounts** on mode select (not an overlay). Back from choose-path uses `closeSubscreen` → returns to `menu` ([`useAppStore`](../../src/renderer/store/useAppStore.ts) `openModeSelect` / `closeSubscreen`).
- **MainMenu fit:** `fitShellPadding = width >= 1024 && height <= 760 ? 8 : 12` + `useFitShellZoom({ padding: fitShellPadding, … })` ([`MainMenu.tsx`](../../src/renderer/components/MainMenu.tsx) ~L70–77).
- **Choose Your Path fit:** `pathFitPadding = vpW >= 1024 && vpH <= 760 ? 8 : 14` + same hook ([`ChooseYourPathScreen.tsx`](../../src/renderer/components/ChooseYourPathScreen.tsx) ~L121–129).
- **Policy doc:** [Viewport fit UI](../VIEWPORT_FIT_UI.md) already groups these shells and mentions `--menu-*` / `--path-*` locals; it does **not** spell out the TS padding split.

**Intentional differences (out of scope for this task):** MainMenu includes Pixi [`MainMenuBackground`](../../src/renderer/components/MainMenuBackground.tsx) + `UI_ART.menuScene`; Choose Your Path stacks `UI_ART.gameplayScene` + `UI_ART.choosePathScene` (static layers). Do not merge atmosphere stacks—only the **fit-zoom contract** and shared breakpoint inputs.

## Problem

- **Drift risk:** Two hand-maintained padding literals for the same “wide short desktop” case (8) but **12 vs 14** otherwise—unclear if 2px is deliberate (denser path grid) or historical drift.
- **Duplicate predicates:** Both screens recompute phone / narrow short-landscape helpers against the same [`breakpoints`](../../src/renderer/breakpoints.ts) exports; a viewport policy change requires two edits.

## One deliverable (this task)

1. **Introduce a single exported helper** (e.g. in `useFitShellZoom.ts` or a tiny `hubShellFit.ts` next to it) such as `getHubShellFitPadding(viewportWidth, viewportHeight)` **or** named constants `HUB_SHELL_FIT_PADDING_DEFAULT` / `HUB_SHELL_FIT_PADDING_SHORT_DESKTOP` used by both screens—**pick one padding pair** (either preserve current behavior with a code comment explaining 12 vs 14, or unify to one value after quick visual check on 1280×720 and 844×390).
2. **Wire** `MainMenu` and `ChooseYourPathScreen` to use that helper/constants only (no duplicated ternary).
3. **Optional one-line** note in [VIEWPORT_FIT_UI.md](../VIEWPORT_FIT_UI.md) under the shell section pointing at the shared helper (authoritative for “menu hub family” padding).

## Verification

- `yarn test` / targeted tests if a pure function is extracted (reuse [`computeFitShellZoomFactor`](../../src/renderer/hooks/useFitShellZoom.ts) patterns).
- Manual or Playwright smoke: main menu and choose-path at **1280×720** and **narrow phone**—no vertical overflow on `[data-app-scrollport]` (existing e2e philosophy in [VIEWPORT_FIT_UI.md](../VIEWPORT_FIT_UI.md)).
