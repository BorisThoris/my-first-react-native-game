# REF-008 — Align imperative `notify*` bridge with store `show*` options

> **Rollup:** Addressed in codebase per [COMPLETION.md](./COMPLETION.md) (evidence row **001–010**).

## Summary

The Zustand store exposes richer helpers (`showSuccess` / `showInfo` with optional `stackKey`, `surface`, `ariaLive`; `showAchievement`) than the module-level `notificationBridge` (`notifyError`, `notifyWarning`, `notifySuccess`, `notifyInfo`) and its `GlobalNotificationHandler` type. Non-React or shared code that must use the imperative API cannot coalesce toasts or pick surfaces without reaching past the bridge.

## Context (authoritative)

- Store API: [`packages/notifications/src/notificationStore.ts`](../../packages/notifications/src/notificationStore.ts) — `showSuccess`, `showInfo`, `showAchievement`, and `addNotification` metadata (`NotificationMeta`, `NotificationSurface`).
- Bridge + registration: [`packages/notifications/src/notificationBridge.ts`](../../packages/notifications/src/notificationBridge.ts), [`packages/notifications/src/NotificationHost.tsx`](../../packages/notifications/src/NotificationHost.tsx) (`setGlobalNotificationHandler` passes only the four `show*` methods with narrow typings).
- Wiki: [TOOLING.md § `@cross-repo-libs/notifications`](../internal-wiki/TOOLING.md), [ARCHITECTURE.md § Notifications](../internal-wiki/ARCHITECTURE.md).

## Proposed refinement (single task)

1. **Widen `GlobalNotificationHandler`** so its method signatures match the corresponding store methods (at minimum optional third-argument `options` for `showSuccess` and `showInfo` where the store already supports them; include `showAchievement` on the handler if a new `notifyAchievement` is added).
2. **Extend `notifySuccess` / `notifyInfo`** (and add `notifyAchievement` if desired) to accept the same optional options objects and forward them to the handler; keep backward-compatible defaults.
3. **Update `NotificationHost`** `useEffect` that calls `setGlobalNotificationHandler` to register the expanded surface (store methods already implement behavior).
4. **Tests:** extend [`packages/notifications/src/notificationBridge.test.ts`](../../packages/notifications/src/notificationBridge.test.ts) (and any host integration test if needed) so forwarded arguments match the store.

## Acceptance criteria

- TypeScript consumers see bridge + handler types consistent with `NotificationStoreState` for the covered helpers.
- Existing call sites that only pass `(message, duration?)` remain valid.
- Unit tests cover at least one options forward path (e.g. `stackKey` or `surface` on success).

## Out of scope (for this item)

- Changing confirm-dialog semantics or CSS tokens; focus is API/type parity only.
