# REF-016 — Web gameplay SFX (`gameSfx.ts`) and `useAppStore` wiring

**Category:** audio / renderer  
**Priority:** P2 (polish & architecture clarity; not blocking core loop)  
**Status:** Done — [`gameSfx.ts`](../../src/renderer/audio/gameSfx.ts), [`useAppStore.ts`](../../src/renderer/store/useAppStore.ts); [COMPLETION.md](./COMPLETION.md) (**011–020**). “Candidate refinements” below are optional polish.

## Scope

Single slice: **synthesized Web Audio SFX** in `src/renderer/audio/gameSfx.ts` and **every call site** from `src/renderer/store/useAppStore.ts` (Zustand). No other modules import `gameSfx` today.

## Authoritative code map

| Symbol | Location | Role |
|--------|----------|------|
| `getAudioContext`, `resumeAudioContext` | `gameSfx.ts` | Lazy singleton `AudioContext`; resume if suspended (autoplay policy). |
| `sfxGainFromSettings(master, sfx)` | `gameSfx.ts` | `clamp01(master) * clamp01(sfx)` — effective linear gain. |
| `playFlipSfx`, `playMatchSfx`, `playMismatchSfx`, `playResolveSfx` | `gameSfx.ts` | Oscillator tones; `playResolveSfx` picks match vs mismatch from `RunState` stat deltas. |
| `sfxGainFromStore()` | `useAppStore.ts` (module closure) | Reads `useAppStore.getState().settings` and delegates to `sfxGainFromSettings`. |
| `applyResolveBoardTurn` | `useAppStore.ts` | `resolveBoardTurn` → `resumeAudioContext` → `playResolveSfx` → `applyResolvedRun`. |
| `pressTile` | `useAppStore.ts` | On successful flip: `resumeAudioContext` → `playFlipSfx(sfxGainFromStore())` (normal path + gambit third-flip path). |

## Behavior summary

1. **Flip feedback** — Plays **synchronously** inside `pressTile` when `flippedTileIds` count increases (standard flips and **gambit third pick** branch). User input still counts as a gesture for `AudioContext` in typical browsers.

2. **Match / mismatch feedback** — Plays from **`applyResolveBoardTurn`**, which runs when `scheduleResolveTimer` fires (or immediately if resolve delay ≤ 0). So resolve SFX is **deferred** relative to the second tile press by `resolveRemainingMs` (and `resolveDelayMultiplier` in settings flows into run state). **Not** tied to the flip moment.

3. **Volume** — Only `Settings.masterVolume` and `Settings.sfxVolume` (`contracts.ts`) affect SFX. There is no separate music pipeline in this module.

4. **Duplication** — `applyResolveBoardTurn` uses `sfxGainFromSettings(settings.masterVolume, settings.sfxVolume)` from a fresh `getState()`; flip path uses `sfxGainFromStore()` — same numbers, two styles.

## Findings / gaps

- **Store ↔ audio coupling** — `useAppStore.ts` imports renderer audio directly. Tests do not mock `gameSfx`; behavior relies on try/catch around `new AudioContext()` and early returns when `gain` is tiny. No dedicated unit tests for SFX or gain math.
- **Async resolve + autoplay** — Resolve tones run from a **timer**, not a direct user event. In strict environments, if the context were not running, match/mismatch could be silent while flips still work after first gesture; worth validating on target WebView/Electron builds if audio feels inconsistent.
- **Coverage** — Powers (shuffle, destroy, peek, etc.) and navigation do not use `gameSfx`; only flip + resolve resolution paths.

## Candidate refinements (triage)

1. **Unify gain** — Use `sfxGainFromStore()` inside `applyResolveBoardTurn` for one code path (or pass `gain` as an argument from a single helper) to avoid drift.
2. **Optional injection** — For testability or future native audio, abstract “play flip / play resolve” behind a small interface set on the store at bootstrap (heavier change; only if product needs it).
3. **Document timing** — One sentence in `gameSfx.ts` or store comment: resolve SFX fires after resolve delay, not on flip.
4. **QA note** — Add a manual or e2e checklist row: mute SFX / master sliders, gambit third flip, zero-delay resolve.
