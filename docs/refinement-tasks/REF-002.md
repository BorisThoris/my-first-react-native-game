# REF-002: Coerce remaining `Settings` string unions in `normalizeSaveData`
> **Rollup:** Addressed in codebase per [COMPLETION.md](./COMPLETION.md) (evidence row **001–010**).


## Problem

`normalizeSaveData` already normalizes a subset of `Settings` (`boardScreenSpaceAA`, `graphicsQuality`, `boardBloomEnabled`, `pairProximityHintsEnabled`, `cameraViewportModePreference`). Other fields that are narrow unions on `Settings`—for example `displayMode`, `weakerShuffleMode`, `boardPresentation`—are merged as-is. JSON from `localStorage` or hand-edited saves can contain strings that are not members of those unions at runtime while TypeScript still assumes the contract holds.

## Proposed work

- Add the same string-literal allowlist pattern used for AA / graphics / camera for every remaining `Settings` field that is a discriminant union. Unknown or legacy values fall back to `DEFAULT_SETTINGS` for that key.
- Optionally extract small `normalizeXxx(value, fallback)` helpers beside `normalizeSaveData` for readability.
- Extend `save-data.test.ts` with bogus-string cases (mirror existing `cameraViewportModePreference` tests) for at least one additional union (e.g. `weakerShuffleMode`).

- Invalid persisted values for each covered union field are replaced by defaults; valid values round-trip unchanged.
