## Summary

Previously, `resumeRunWithTimers` used **truthy** checks on `memorizeRemainingMs` and `debugRevealRemainingMs`, so **`0` ms** skipped `scheduleMemorizeTimer` / `scheduleDebugRevealTimer` and could leave memorize or debug peek stuck at zero after resume. **Now** both paths use explicit `!== null` gating via [`runTimerResumeConditions.ts`](../../src/renderer/store/runTimerResumeConditions.ts). `resolveRemainingMs` was already correct.

## Evidence

`RunTimerState` allows numeric zero; it is not “no timer” (that is `null`):

```263:267:src/shared/contracts.ts
export interface RunTimerState {
    memorizeRemainingMs: number | null;
    resolveRemainingMs: number | null;
    debugRevealRemainingMs: number | null;
```

**Fixed:** `resumeRunWithTimers` now gates memorize/debug via `!== null` semantics through `shouldScheduleMemorizeTimerOnResume` / `shouldScheduleDebugRevealTimerOnResume` (see [`runTimerResumeConditions.ts`](../../src/renderer/store/runTimerResumeConditions.ts)). Resolve path was already correct.

`freezeRun` can persist `0` via `getRemainingMs` (clamped to `Math.max(..., 0)`):

```396:415:src/renderer/store/useAppStore.ts
const freezeRun = (run: RunState): RunState => {
    const pausedRun = pauseRun(run);

    return {
        ...pausedRun,
        timerState: {
            ...pausedRun.timerState,
            memorizeRemainingMs:
                run.status === 'memorize'
                    ? getRemainingMs(memorizeTimer, run.timerState.memorizeRemainingMs)
                    : pausedRun.timerState.memorizeRemainingMs,
            resolveRemainingMs:
                run.status === 'resolving'
                    ? getRemainingMs(resolveTimer, run.timerState.resolveRemainingMs)
                    : pausedRun.timerState.resolveRemainingMs,
            debugRevealRemainingMs: run.debugPeekActive
                ? getRemainingMs(debugRevealTimer, run.timerState.debugRevealRemainingMs)
                : pausedRun.timerState.debugRevealRemainingMs
        }
    };
};
```

The schedulers already centralize “0 means complete now”:

```312:337:src/renderer/store/useAppStore.ts
function scheduleMemorizeTimer(duration: number): void {
    clearMemorizeTimer();

    if (duration <= 0) {
        const { run } = useAppStore.getState();

        if (run && run.status === 'memorize') {
            useAppStore.setState({ run: finishMemorizePhase(run) });
        }

        return;
    }
    // ...
}
```

## Implementation

- Memorize/debug resume conditions aligned with resolve: `shouldScheduleMemorizeTimerOnResume` / `shouldScheduleDebugRevealTimerOnResume` in [`runTimerResumeConditions.ts`](../../src/renderer/store/runTimerResumeConditions.ts); `resumeRunWithTimers` in `useAppStore.ts` calls `schedule*Timer` when those return true.
- **`start*` / transition paths:** `useAppStore` now uses `memorizeRemainingMs !== null` (and `debugRevealRemainingMs !== null` where applicable) before `scheduleMemorizeTimer` / `scheduleDebugRevealTimer`, matching the `0` ms immediate-completion semantics of the schedulers.

## Acceptance criteria

- [x] Resuming from pause or closing meta overlay with **memorize remaining exactly `0`** advances memorize (same observable behavior as `scheduleMemorizeTimer(0)`).
- [x] Same for **debug reveal** when `debugRevealRemainingMs === 0` and peek is active.
- [x] No regression: `null` still means “no active countdown / not applicable” for each field per existing game logic.
- [x] Focused unit tests: [`runTimerResumeConditions.test.ts`](../../src/renderer/store/runTimerResumeConditions.test.ts) (REF-004); `resumeRunWithTimers` + `freezeRun` integration left optional if behavior is fully covered by predicates + schedulers.
