## Scope (one task)

Single research slice covering the **top HUD deck** ([`GameplayHudBar.tsx`](../../src/renderer/components/GameplayHudBar.tsx)) and the **left vertical rail** ([`GameLeftToolbar.tsx`](../../src/renderer/components/GameLeftToolbar.tsx)), including how [`GameScreen.tsx`](../../src/renderer/components/GameScreen.tsx) mounts them, shared presentation in [`GameScreen.module.css`](../../src/renderer/components/GameScreen.module.css), and documented parity gaps. These two surfaces are intentionally separate components but share **camera viewport mode** styling and **stacking** concerns on narrow layouts.

## Executive summary

- **`GameplayHudBar`** renders the in-run **header** (`<header>`, `data-testid="game-hud"`): primary stat row (floor hex, lives, shards, centered score, daily/parasite right column) and a **secondary context strip** (mode label, context/mutator chips, compact `statRail`). It owns `MUTATOR_HUD_LABELS`, inline mutator chip glyphs, and an **`aria-live="polite"`** region for `politeHudAnnouncement` (HUD-015). It returns `null` if `run.board` is missing.
- **`GameLeftToolbar`** renders an **`<aside aria-label="Game actions">`** with up to four vertical **toolbars**: primary controls (fit, settings, codex, inventory, main menu + dev reveal when enabled), optional rules toggle, conditional **board powers** row, and **resolve/undo** while `run.status === 'resolving'`. It uses **`toolbarRoving`** (`syncVerticalToolbarTabIndices`, `handleVerticalToolbarKeyDown`) per toolbar `role="toolbar"`.
- **`cameraViewportMode`** toggles **`mobileCameraHud`** vs **`mobileCameraLeftToolbar`** classes so the **left rail stays above the HUD** (z-index) and typography/spacing tighten on mobile camera layout—see CSS comments near `.mobileCameraLeftToolbar` / `.mobileCameraHud`.

## Behavior notes (code-aligned)

### GameplayHudBar

- **Wings / e2e hooks:** `data-testid="hud-wing-left"`, `"hud-wing-center"`, `"hud-wing-right"` — **HUD-018 / QA-003** in [`e2e/navigation-flow.spec.ts`](../../e2e/navigation-flow.spec.ts) and [`e2e/mobile-layout.spec.ts`](../../e2e/mobile-layout.spec.ts) warn to update together if the HUD splits.
- **PLAY-003:** Comment block (~L179) documents IA: primary row = “slim strip” read; mode/mutator/stat rail on **second row** on wide layouts; narrow stacks primary then context (flyout rejected for mutator visibility).
- **PLAY-004:** Comment (~L253) — reference mock three hearts vs `MAX_LIVES` (5); live UI renders five slots + `run.lives / MAX_LIVES` readout.
- **Mutators:** `score_parasite` is filtered from chip row when active (shown in dedicated parasite module). Context chips can add gauntlet, scholar, shuffle-tax.

### GameLeftToolbar

- **Primary rail order (catalog / PLAY-001):** fit → settings → codex → inventory → main menu (abandon opens confirm in `GameScreen` via `onRequestAbandonRun`). Pause is **not** on the rail.
- **Powers block:** Gated by `showBoardPowerBar`; includes shuffle (with optional `TileBoard` animation hook), row shuffle cluster, pin, destroy, peek, flash pair (conditional), stray remove.
- **Test IDs:** `game-toolbar-codex`, `game-toolbar-inventory` (and others per TASKS / e2e references); not every control has a `data-testid`.

### Layout interaction (HUD vs rail)

- CSS explicitly stacks **left toolbar above HUD** in mobile camera mode so rail taps are not blocked by the HUD overlay (see `.mobileCameraLeftToolbar` z-index vs `.mobileCameraHud`; cross-check [`docs/new_design/TASKS/TASK-020-endproduct-screenshot-audit-and-captures.md`](../../docs/new_design/TASKS/TASK-020-endproduct-screenshot-audit-and-captures.md)).
