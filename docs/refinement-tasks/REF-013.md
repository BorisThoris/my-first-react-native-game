# REF-013 — SettingsScreen: unsaved draft discarded on Back

| Field | Value |
|--------|--------|
| **ID** | REF-013 |
| **Scope** | `SettingsScreen` |
| **Category** | UX / settings persistence |
| **Priority** | Medium |
| **Status** | Addressed — current draft + Back behavior documented; confirm-on-dirty-back optional polish, not shipped. [COMPLETION.md](./COMPLETION.md) (**011–020**). |

## Observation

`SettingsScreen` keeps edits in local `draft` state and writes to disk only when the user clicks **Save**. **Back** always calls `closeSettings` from the store and never checks `isDirty`, so any unsaved changes are **silently discarded**. There is no confirmation, second chance, or auto-save.

## Evidence

- Draft vs persisted settings and dirty flag: `draft`, `isDirty`, `handleSave` / `patchSettings` in [`src/renderer/components/SettingsScreen.tsx`](../../src/renderer/components/SettingsScreen.tsx) (e.g. draft state around lines 257–273, 371–389, footer 778–795).
- **Back** wires directly to `closeSettings` with no guard: same file, footer `UiButton` `onClick={closeSettings}`.
- Store `closeSettings` only switches `view` (and resumes the run when opened from play): [`src/renderer/store/useAppStore.ts`](../../src/renderer/store/useAppStore.ts) (`closeSettings` ~883–901). It does not receive or merge draft settings.

## User impact

Players who tweak sliders/toggles, forget to hit **Save**, then press **Back** (menu or in-run modal) lose changes without feedback. This is easy to hit on long sessions or when exploring the shell.

## Recommended fix (one deliverable)

Implement an **explicit decision** when closing with a dirty draft: e.g. confirm dialog (**Save / Discard / Cancel**), or **auto-persist** draft on **Back** (product call). Whichever approach ships, align **in-run modal** and **full-page** settings (`presentation` `'modal'` vs `'page'`) and add a **focused test** (unit or e2e) that reproduces: dirty draft → Back → expected behavior (no silent loss unless user confirms discard).
