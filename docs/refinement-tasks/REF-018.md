## Question

How do in-game `AchievementId` values map to Steam Stats & Achievements **API Names**, and what must stay in sync when achievements change?

## Authoritative code

| Location | Role |
|----------|------|
| `src/shared/contracts.ts` | `AchievementId` union (**5** literals). |
| `src/main/steam.ts` | `STEAM_ACHIEVEMENT_API_NAME`: `Record<AchievementId, string>` via `satisfies` — **one string per id**, passed to `client.achievement.activate(apiName)`. |
| `src/main/ipc.ts` | Handler `desktop:unlock-achievement`: calls `persistence.unlockAchievement` **first**, then `steamAdapter.unlockAchievement`. |
| `src/shared/save-data.ts` | `ACHIEVEMENT_IDS` array — same **5** ids, order used for initial state. |
| `src/shared/mechanics-encyclopedia.ts` | `ACHIEVEMENT_CATALOG` — player-facing copy; test ensures catalog covers every `AchievementId`. |

## Findings

1. **Mapping is explicit and type-checked.** `STEAM_ACHIEVEMENT_API_NAME` is declared `as const satisfies Record<AchievementId, string>`, so adding or renaming an `AchievementId` without updating this object is a **TypeScript error**.

2. **Current mapping is identity.** Each key’s value equals the `AchievementId` string (e.g. `ACH_FIRST_CLEAR: 'ACH_FIRST_CLEAR'`). Comments in `steam.ts` state that Partner **API Names** are expected to match; if the Steamworks dashboard uses different strings, **only the map values** should change — not `AchievementId` or save data keys.

3. **Persistence is independent of Steam.** Unlock IPC always writes the achievement into save data before calling Steam; the mock adapter (`steamworks` init failure) still keeps local unlocks; Steam activation is best-effort (`boolean`, errors logged).

4. **Single call site.** Real adapter uses `STEAM_ACHIEVEMENT_API_NAME[achievementId]` only inside `unlockAchievement`; no duplicate mapping elsewhere in `src/main`.

## Gaps / verification (outside repo)

- **Steam Partner dashboard:** Confirm each achievement’s **API Name** matches the string values in `STEAM_ACHIEVEMENT_API_NAME` (count **5**). [internal snapshot](../internal-wiki/APP_ANALYSIS_SNAPSHOT_2026-04-17.md) records parity as verified at analysis time; re-check after any Partner rename.

## Recommended refinements (optional)

| Action | Rationale |
|--------|-----------|
| When adding an achievement: extend `AchievementId` → fix compile errors in `STEAM_ACHIEVEMENT_API_NAME`, `ACHIEVEMENT_IDS`, catalog, and unlock evaluation — **no separate “Steam checklist”** beyond the `satisfies` map. | Process is already enforced by types. |
| If Partner names diverge, document the mapping in a one-line comment per entry or in `mechanics-encyclopedia` only if players need to see Steam’s internal name (usually not). | Reduces silent mismatch risk. |

## Conclusion

**No code defect identified.** The mapping is centralized, fully covering `AchievementId`, and compile-safe. The only operational risk is **dashboard API Name drift** vs. the string literals in `STEAM_ACHIEVEMENT_API_NAME` — fix by updating values in `steam.ts` only.
