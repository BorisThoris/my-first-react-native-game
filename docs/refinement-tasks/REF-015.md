# REF-015 — InventoryScreen + CollectionScreen (single meta-surface pass)

**Status:** Baseline shipped — evidence in [COMPLETION.md](./COMPLETION.md) (≈ **031–040** / meta UI). Remaining items below are **optional polish**, not release gates.  
**Priority:** P1 (polish; pairs with design tasks META-006 / META-007)  
**Category:** Meta UI — read-only archive + active-run loadout  
**Last triage:** 2026-04-17 (REF-100)

## Summary

Ship one coordinated refinement across **Collection** (persistent save/progress) and **Inventory** (active-run snapshot) so both screens feel like the same “archive desk” family: shared information hierarchy, comparable card density for list-like content, and aligned use of `MetaFrame` / hero panels. Today **Collection** leans on grids, tier tints, and `MetaFrame` on key sections; **Inventory** is mostly key–value rows and plain lists with stronger treatment only in the in-run desk overlay.

## Current behavior (facts)

- **Collection:** `saveData`-driven read-only UI — achievements grid (`ACHIEVEMENT_IDS`), full relic catalog with pick-count tier classes (`relicTierLatent` / `Known` / `Forged`), bests/daily stats, symbol gallery bands. In-page TOC anchors: achievements, relics, bests, daily, symbols. `MetaFrame` wraps achievements and relics sections; other sections are `Panel` only.
- **Inventory:** `run`-driven read-only UI — run snapshot, relic/mutator title lists, charges/tokens KV, contract flags. Same TOC pattern. `MetaFrame` wraps run hero + empty state; relic/mutator/charges/contract are default `Panel`s. **`stackedOnGameplay`** switches shell class (`shellInRunModal`), panel chrome (`.inRunHeroPanel` / `.inRunPanel`), and `ScreenTitle` to `h2` for a single document `h1` on `GameScreen`.
- **E2E:** Collection and inventory (menu + in-run) covered in [`e2e/navigation-flow.spec.ts`](../../e2e/navigation-flow.spec.ts), visual steps in [`e2e/visualScenarioSteps.ts`](../../e2e/visualScenarioSteps.ts) (`01b-collection`, `01c-inventory-empty`, `01d-inventory-active`).

## Gaps (why one pass)

1. **Visual language split:** Collection relics use card grids + tier borders; Inventory relics/mutators are bulleted titles — players scanning both screens see **two different metaphors** for “relic identity.”
2. **META-007 alignment:** Task table calls for a **loadout board** (slots, icons, charge pips). Inventory remains **KV + lists**; refinement should either narrow the task wording or move UI toward that spec without duplicating Codex depth.
3. **META-006 alignment:** Achievement/relic cards are **CSS-only** medallions; long achievement lists are a single grid — **performance and scroll ergonomics** on low-end webviews deserve a single decision (virtualization vs. collapsible bands) shared with Inventory if list density grows.
4. **Frame parity:** Only some Collection sections use `MetaFrame`; Inventory uses it for the hero (and empty) but not for relic/mutator blocks — **inconsistent “forged plate” emphasis** across the two screens.
5. **Menu vs in-run:** Inventory gains bespoke gold-tinted chrome in `.inRunInventoryShell`; Collection has no in-run counterpart — acceptable, but the pass should document **when** menu-path Collection should echo desk styling (probably never unless Collection is opened over gameplay later).

## Proposed refinement (single deliverable)

1. **Unify “catalog row” presentation:** Introduce a small shared presentational pattern (or one module) for “title + short description + optional meta line” used for **Inventory relic/mutator lines** and optionally **Collection relic cards** on narrow viewports — so both screens share typography and spacing tokens even if Collection keeps grids on desktop.
2. **Raise Inventory relic/mutator to framed rows or compact cards** (icons optional / placeholder) so the screen matches Collection’s relic tier story **without** turning Inventory into a second Codex — link titles to `RELIC_CATALOG` / `MUTATOR_CATALOG` fields already in code.
3. **Apply `MetaFrame` + `Panel` consistently** to the first “content” stack on each screen (or document an explicit rule: hero + first detail band only) so TOC anchors land on visually consistent blocks.
4. **Verify a11y:** Preserve `aria-label` regions, heading order (`stackedOnGameplay` → `h2`), and keyboard scroll-to-anchor for TOC links; extend e2e only if new interactive controls appear.

## Acceptance criteria (optional polish)

Revisit when touching Collection/Inventory chrome; not tracked as blockers for `yarn verify`.

- **Visual:** Menu-path Collection and menu-path Inventory read as **one family** (spacing, section headers, card/list treatment) per COMPONENT_CATALOG meta table; in-run Inventory keeps **desk overlay** distinct but not clashing with new shared tokens.
- **Functional:** Still strictly read-only; no new store fields required for v1 unless explicitly needed for display (e.g., icon ids already in catalog).
- **Regression:** Existing e2e scenarios for collection + inventory navigation and `data-testid` hooks (`collection-meta-frame-*`, `inventory-meta-frame-*`) still pass; update snapshots if intentional.
- **Docs:** Brief note in COMPONENT_CATALOG or TASKS_META_AND_SHELL that META-006/META-007 partial scope is satisfied or resliced — **one** authoritative paragraph, no duplicate narrative.

## Dependencies / risks

- **Depends on:** META-003 primitives (`MetaFrame`, theme tokens) already in tree; no blocker.
- **Risk:** Over-building Codex-like detail in Inventory — keep **run snapshot** focus (charges, contract) obvious above the fold.
- **Out of scope for REF-015:** Codex content, GameScreen HUD, save format changes, Steam/stat sync.
